# 智能客服 Agent 架构演进路线

本文档沉淀智能客服项目后续可做项和优化方向，用于指导后续迭代。当前目标不是一次性重构，而是在现有架构上逐步增强客户自助解决能力，尽可能减少人工干预。

## 1. 当前项目现状

已具备的能力：

- 用户端客服对话界面。
- FastAPI 后端会话接口。
- 意图识别：规则优先，模型兜底。
- 知识库 RAG：用于平台规则、教程、故障排查类问题。
- Mock 业务查询：提现订单、充值 TxID 查询。
- 多轮补问：提现订单号、充值 TxID 缺失时提示补充。
- 结构化下一步动作：通过 `next_action` 告诉前端期望用户补充什么。
- 前后台 API 初步拆分：客户交互接口与运营统计接口分离。
- 运营 trace：记录意图来源、路由、处理结果等统计数据。
- CLI 和 ops API：支持查看对话 trace 汇总。

主要缺口：

- 缺少明确的对话状态机。
- 实体抽取还散落在意图识别和业务规则中。
- 多轮任务流只覆盖少量场景，缺少统一的状态流转模型。
- 评估体系偏重意图识别，缺少实体、补问、完成率等指标。
- 知识检索主要依赖现有 RAG，还没有混合检索和结果融合策略。
- 人工兜底目前只记录候选，缺少系统化复盘和优化闭环。

## 2. 参考方案中可吸收的设计

参考方案中的完整链路是：

```text
用户输入
-> 文本预处理
-> 意图识别
-> 实体抽取
-> 对话状态管理
-> 知识检索 / 业务查询 / 补问
-> 回复生成
-> 用户反馈
-> 评估与迭代
```

适合吸收到本项目的部分：

1. 输入预处理
   - 统一清理空白字符、大小写、常见符号。
   - 对订单号、TxID、币种、网络等字段做标准化。
   - 保留用户原文，标准化结果只用于识别和查询。

2. 意图识别与实体抽取拆分
   - 意图识别负责判断用户要解决什么问题。
   - 实体抽取负责识别订单号、TxID、币种、网络、时间等关键字段。
   - 两者结果共同决定后续处理路径。

3. 对话状态管理
   - 将当前 `next_action` 扩展为轻量状态模型。
   - 明确区分待补充、处理中、已完成、无法识别、兜底候选等状态。
   - 避免前端依赖文案判断流程。

4. 混合检索
   - 业务查询优先处理确定性问题。
   - 知识类问题进入 RAG。
   - 后续可增加关键词召回、类目过滤、向量检索融合排序。

5. 评估体系
   - 不只评估意图识别准确率。
   - 增加实体抽取准确率、补问成功率、多轮完成率、人工兜底候选率。
   - 将线上 trace 反向转化为评估样本。

6. 生产稳定性
   - 记录关键路径耗时。
   - 对知识检索类结果考虑缓存。
   - 对业务查询、模型调用、RAG 调用分别记录失败类型。

## 3. 核心业务边界

### 提现问题

提现客服主流程聚焦平台侧是否放行，而不是链上排查。

业务原则：

- 已完成、已广播或已上链的提现，链上状态通常透明，不进入客服排查主流程。
- 用户可通过 TxID、区块浏览器、目标地址和网络自行核对链上进度及接收方入账规则。
- 平台客服主要确认平台侧订单是否已经放行。
- 真正需要客服介入的是平台侧未放行、审核中、处理中、风控限制、合规检查、安全限制、违规提示、拒绝或卡住等情况。
- 无提现订单号时，优先补问提现订单号。
- 有提现订单号时，优先查询平台侧订单状态。
- 对具体风控、违规或合规原因不做猜测，回复应以页面提示或平台审核结果为准。

建议意图边界：

- `status_query`：用户提供明确提现订单号，查询平台侧订单状态。
- `missing_arrival`：用户表达提现审核、风控、限制、卡住、不放行等平台侧问题，但缺少订单号。
- `onchain_status`：用户明确表达提现已完成、已广播或已上链，但目标钱包未到账；此类问题给出链上透明说明，不补问订单号。

### 充值问题

充值未到账仍保留链上排查路径。

业务原则：

- 充值方向需要确认 TxID、币种、网络、充值时间和页面提示。
- 缺少 TxID 时优先补问充值 TxID。
- TxID 未匹配到平台记录时，可继续收集币种、网络、充值时间和页面提示，用于人工兜底或后续排查。
- 不要把提现“链上透明”的处理原则直接套用到充值问题。

## 4. 推荐演进路线

### 阶段 1：轻量对话状态模型

目标：

- 把当前 `next_action` 升级为更完整的对话状态描述。
- 保持不改数据库，先在服务层计算并通过 API 返回。

建议状态：

```text
idle
awaiting_withdrawal_order_id
awaiting_deposit_txid
awaiting_problem_description
processing_business_query
answer_from_knowledge
completed
unknown
manual_fallback_candidate
```

建议输出结构：

```json
{
  "state": "awaiting_withdrawal_order_id",
  "expected_input": "withdrawal_order_id",
  "missing_fields": ["order_id"],
  "manual_fallback_candidate": false
}
```

验收标准：

- 前端输入提示完全由结构化字段驱动。
- 旧消息没有结构化字段时仍兼容现有 fallback。
- 提现、充值、unknown、human request 均有明确状态。

### 阶段 2：实体抽取模块

目标：

- 将实体识别从意图规则中拆出，形成独立模块。
- 先做规则抽取，不引入训练模型。

首批实体：

- `withdrawal_order_id`
- `deposit_txid`
- `coin`
- `network`
- `time_hint`

处理原则：

- 明确编号直接抽取。
- pending 场景下允许纯数字补全前缀。
- 非 pending 场景不猜测纯数字含义。
- 识别结果和用户原文分开保存，避免篡改聊天记录。

验收标准：

- 新增实体抽取测试集。
- 提现订单号、充值 TxID、纯数字 pending 补全都有测试。
- 实体抽取结果进入 trace，便于后续统计。

### 阶段 3：多轮任务状态流

目标：

- 将“缺字段 -> 补问 -> 用户补充 -> 查询 -> 完成/失败/兜底候选”固化为统一流程。

首批任务：

- 查询提现订单状态。
- 查询充值 TxID 状态。
- 充值未到账排查。
- 提现平台侧审核、风控、限制和不放行状态排查。
- 提现已完成、已广播或已上链后的链上透明说明。
- 身份认证失败排查。
- 账户安全风险处理。

验收标准：

- 每个任务都有 required fields。
- 缺字段时只补问必要字段。
- 用户补充字段后能继续原任务。
- 超时后关闭旧任务，按新问题处理。

### 阶段 4：评估体系扩展

目标：

- 从只看意图准确率，扩展到完整客服链路指标。

建议指标：

- 意图识别准确率。
- 实体抽取 Precision / Recall / F1。
- missing field 判断准确率。
- next action 准确率。
- 多轮任务完成率。
- unknown 占比。
- manual fallback candidate 占比。
- RAG 命中率。
- 业务查询完成率。

验收标准：

- 评估脚本可以输出上述核心指标。
- 评估样本覆盖正常问法、口语问法、缺字段问法、干扰问法。
- trace 中的失败样本可以反向加入评估集。

### 阶段 5：混合检索优化

目标：

- 提升知识类问题命中率和答案稳定性。

推荐顺序：

1. 类目过滤：根据意图 category 限定知识库范围。
2. 关键词召回：补充向量召回漏掉的精确词。
3. 向量召回：保留现有语义检索。
4. 结果融合：综合向量分、关键词分、类目匹配分。
5. 答案引用校验：继续要求回答基于检索结果。

验收标准：

- 常见充值、提现、认证、账户安全问题命中正确文章。
- 无关问题不会强行返回知识库答案。
- 检索结果可解释，能看到命中文章和分数来源。

### 阶段 6：运营指标沉淀

目标：

- 运营侧能看清楚哪些问题被自动解决，哪些问题导致兜底候选。

建议指标：

- 总会话轮次。
- 按意图来源统计：rule / model / fallback。
- 按路由统计：business_query / knowledge_rag / unknown / out_of_scope。
- 按处理结果统计：业务查询成功、缺字段、RAG 回答、未知、兜底候选。
- 高频 unknown 用户原文样本。
- 高频 manual fallback candidate 样本。
- 补问后完成率。

验收标准：

- CLI 能查看核心指标。
- ops API 能返回结构化统计。
- 后续后台页面只消费 ops API，不混入客户交互 API。

### 阶段 7：生产稳定性与性能优化

目标：

- 在业务链路稳定后，再优化性能和可用性。

可选优化：

- 对知识检索结果做短期缓存。
- 对静态 FAQ 或热门问题做缓存。
- 记录模型调用耗时和失败原因。
- 区分业务查询错误、模型错误、检索错误、数据库错误。
- 对外部服务调用设置超时和降级提示。

暂缓事项：

- 暂不引入复杂异步编排框架。
- 暂不引入训练模型。
- 暂不拆物理服务，先保持 API 和目录分离。

## 5. 暂不做的事项

以下事项当前不优先：

- 不引入 BERT、BiLSTM、CRF 等训练模型。
- 不新增机器学习训练链路。
- 不引入新的 Web 框架。
- 不大规模重构现有 service / repository / api 分层。
- 不把人工流程展示给用户。
- 不先做权限系统。
- 不立即拆成多个物理服务。
- 不对业务查询结果做不安全缓存。

## 6. 优先级清单

P0：

- 轻量对话状态模型。
- 实体抽取模块。
- 多轮补问和任务完成闭环。
- next action / conversation state 结构化输出。

P1：

- 评估体系扩展。
- trace 样本复盘。
- unknown 和 manual fallback candidate 分析。
- ops 指标完善。

P2：

- 混合检索。
- 知识库类目过滤。
- 关键词召回与结果融合。
- 生产缓存和性能优化。

## 7. 下一步建议

下一步优先做“阶段 6：运营指标沉淀”的小步增强，围绕提现平台侧审核、风控、限制和不放行问题补齐统计口径。

建议边界：

- 不改数据库。
- 不改前端页面结构。
- ops API 和 CLI 先消费现有 trace。
- 将 `business_withdrawal_pending_review`、`withdrawal_onchain_transparent`、`manual_fallback_candidate` 纳入统计。
- 区分平台侧未放行问题和链上透明说明，避免把两类问题混在同一个“提现未到账”指标里。
- 测试覆盖统计结果和空数据场景。

完成后再考虑更细的任务状态流和人工复盘闭环。
